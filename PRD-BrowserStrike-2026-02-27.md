# PRD: BrowserStrike — Браузерный FPS-шутер в стиле CS2

**Версия:** 1.0
**Дата:** 27 февраля 2026
**Автор:** PRD Creation Assistant

---

## 1. Обзор продукта

### 1.1 Описание
BrowserStrike — браузерный мультиплеерный FPS-шутер в стиле Counter-Strike 2, ориентированный на быстрые дуэли 1v1 и 2v2. Игра использует упрощённую low-poly 3D-графику с реалистичной цветовой палитрой, оптимизированную для слабых ПК. Подключение к матчам происходит по коду комнаты, без регистрации — только никнейм.

### 1.2 Проблема
Существующие браузерные шутеры (Krunker.io, Shell Shockers) ориентированы на казуальный геймплей и большие лобби. Нет качественного браузерного аналога для тактических дуэлей в стиле CS2 с раундовой системой и выбором оружия.

### 1.3 Цели продукта
- Доступный через браузер тактический шутер без установки
- Минимальный порог входа: никнейм → код комнаты → играй
- Стабильный сетевой FPS на слабых ПК (целевой минимум: 30 FPS на интегрированной графике)
- Тактический геймплей с раундовой системой и выбором оружия

### 1.4 Целевая аудитория
- Казуальные геймеры, которые хотят быстро поиграть с друзьями без установки
- Фанаты CS2, которые хотят поиграть на рабочем/слабом ПК
- Игроки, предпочитающие короткие сессии (дуэли 1v1, 2v2)

## 2. Основные функции и функциональность

### 2.1 Система лобби и комнат

**Описание:** Игрок может создать или присоединиться к комнате по коду.

**Пользовательский флоу:**
1. Игрок заходит на сайт → вводит никнейм
2. Выбирает "Создать комнату" или "Присоединиться"
3. **Создатель (админ)** настраивает:
   - Режим: 1v1 или 2v2
   - Карту (одна из 3–4 доступных)
   - Количество раундов до победы (например: 5, 7, 10, 13)
4. Система генерирует уникальный код комнаты (6 символов, буквы + цифры)
5. Админ делится кодом с друзьями
6. Присоединившиеся игроки попадают в лобби комнаты
7. Игроки выбирают команду (Team A / Team B)
8. Админ нажимает "Начать игру", когда команды укомплектованы (1v1: по 1 игроку в каждой; 2v2: по 2)

**Acceptance Criteria:**
- Код комнаты уникален и состоит из 6 алфавитно-цифровых символов
- Никнейм: 3–16 символов, латиница + цифры
- Админ не может начать игру, пока команды не укомплектованы
- Если игрок отключается в лобби, его слот освобождается
- Максимум 4 игрока в комнате (2v2)

### 2.2 Движение и управление

**Описание:** Классическое FPS-управление от первого лица.

**Управление:**
| Действие | Клавиша |
|---|---|
| Движение вперёд/назад/влево/вправо | W / S / A / D |
| Прыжок | Space |
| Прицеливание | Мышь (Pointer Lock API) |
| Стрельба | ЛКМ (левая кнопка мыши) |
| Выбор оружия | 1 / 2 / 3 или колёсико мыши |
| Scoreboard | Tab (зажать) |

**Физика движения:**
- Скорость ходьбы: ~250 units/sec (аналог CS2)
- Прыжок: фиксированная высота, без bunny-hop (упрощение)
- Коллизии с геометрией карты (стены, ящики, объекты)
- Гравитация: стандартная

**Acceptance Criteria:**
- Pointer Lock активируется по клику на игровое поле
- Esc — выход из Pointer Lock (пауза/меню)
- Движение плавное на 60 FPS при стабильном соединении
- Игрок не может пройти сквозь стены или объекты

### 2.3 Система оружия

**Описание:** 3 вида оружия с уникальными характеристиками. Выбор оружия происходит перед каждым раундом.

**Арсенал:**

| Оружие | Тип | Урон (тело) | Урон (голова) | Скорострельность | Магазин | Дальность |
|---|---|---|---|---|---|---|
| Desert Eagle | Пистолет | 35 | 70 | Полуавтомат (~2 выстр/сек) | 7 | Средняя |
| SSG-08 | Снайперская | 75 | 150 (мгновенное убийство) | Bolt-action (~1 выстр/1.5 сек) | 10 | Дальняя |
| MP9 | Пистолет-пулемёт | 20 | 40 | Автомат (~10 выстр/сек) | 30 | Ближняя-средняя |

**Механики:**
- Разброс (spread): увеличивается при стрельбе очередями (MP9) и при движении
- Отдача (recoil): визуальная отдача камеры, у каждого оружия своя
- Перезарядка: автоматическая, когда магазин пуст; ручная по клавише R
- Время перезарядки: Deagle 2 сек, SSG-08 2.5 сек, MP9 2.5 сек
- Хэдшоты: множитель x2 к базовому урону

**Выбор оружия перед раундом:**
- 5-секундное окно выбора оружия перед началом каждого раунда
- Интерфейс: 3 карточки оружия с характеристиками
- По умолчанию выбран Desert Eagle
- Если время вышло — применяется последний выбор (или дефолт)

**Acceptance Criteria:**
- Каждое оружие имеет уникальную модель, звук выстрела и анимацию
- Урон рассчитывается на сервере (авторитарная модель)
- Хэдшот регистрируется при попадании в хитбокс головы (отдельный от тела)
- Перезарядка прерывается при смене оружия

### 2.4 Система раундов

**Описание:** Матч состоит из серии раундов. Раунд заканчивается, когда одна команда полностью уничтожена.

**Правила:**
- **1v1:** Раунд заканчивается при смерти одного из игроков
- **2v2:** Раунд заканчивается при смерти обоих игроков одной команды
- Победа в матче: команда первой набирает N раундов (настраивается админом)
- Между раундами — пауза 3 секунды с отображением:
  - Счёт раундов (Team A : Team B)
  - "Раунд выиграла Team A/B"
  - Окно выбора оружия (совмещено с паузой + дополнительные 5 секунд)

**Флоу раунда:**
1. Пауза 3 сек (отображение результатов предыдущего раунда)
2. Выбор оружия — 5 сек
3. Спавн игроков на фиксированных точках своей команды
4. Обратный отсчёт 3–2–1
5. Раунд начинается
6. Раунд заканчивается → переход к шагу 1

**Acceptance Criteria:**
- Мёртвый игрок в 2v2 может наблюдать за живым тиммейтом (spectator camera)
- Таймер раунда (опционально, например 2 минуты; если время вышло — ничья в раунде)
- При достижении победного счёта — экран результатов матча

### 2.5 Карты

**Описание:** 3–4 асимметричные карты в стиле локаций CS2, среднего размера с коридорами и позициями.

**Концепция карт:**

| Карта | Стилистика | Описание |
|---|---|---|
| Warehouse | Склад / промзона | Тёмный склад с ящиками, вилочными погрузчиками, узкие проходы между стеллажами |
| Dust Alley | Пустынный городок | Песочные стены, арки, узкие переулки и одна открытая площадь |
| Office | Офисное здание | Коридоры, комнаты, лестницы между этажами (2 этажа) |
| Trainyard | Ж/д станция | Вагоны, платформы, открытые и закрытые зоны |

**Структура карт:**
- Размер: ~40x40 метров (достаточно для дуэлей, но не пустынно)
- Асимметричные: у каждой стороны своя уникальная зона спавна
- 2–3 ключевых маршрута между зонами спавна
- Укрытия, углы, позиции для тактической игры
- Разная высота (ящики, лестницы, балконы)
- Каждая карта должна быть сбалансирована, несмотря на асимметрию

**Технические требования к картам:**
- Low-poly геометрия (< 10,000 полигонов на карту)
- Текстуры: маленькие (256x256 – 512x512), тайловые
- Статичные light maps (запечённое освещение) для производительности
- Формат: GLTF/GLB (оптимальный для Three.js)
- Коллизии: упрощённая геометрия (bounding boxes, простые меши)

**Acceptance Criteria:**
- Каждая карта загружается менее чем за 5 секунд на среднем соединении
- FPS на карте не падает ниже 30 на интегрированной графике
- Игроки не могут выйти за границы карты
- Точки спавна не имеют прямой видимости друг на друга

### 2.6 HUD (интерфейс в игре)

**Элементы:**
- **Прицел (crosshair):** настраиваемый крестик (цвет, размер, толщина, gap)
- **HP:** полоска или число (100/100) — внизу по центру
- **Оружие:** текущее оружие + патроны (текущие / в магазине) — внизу справа
- **Счёт раундов:** Team A : Team B — сверху по центру
- **Таймер раунда:** обратный отсчёт — сверху по центру
- **Килфид (kill feed):** лента убийств — справа сверху, последние 5 событий
- **Scoreboard (Tab):** таблица с никнеймами, убийствами, смертями, пингом — по центру экрана при зажатом Tab
- **Индикатор урона:** красная виньетка при получении урона с указанием направления

**Acceptance Criteria:**
- HUD не закрывает критические зоны обзора
- Настройки прицела сохраняются в localStorage
- Scoreboard обновляется в реальном времени
- Килфид показывает: "[Никнейм1] 🔫 [Никнейм2]" с иконкой оружия

### 2.7 Звуковая система

**Описание:** Звуки дополняют геймплей и помогают определять позицию противников.

**Звуки:**
- Выстрелы: уникальный звук для каждого оружия
- Шаги: звук шагов с базовой пространственной аудиосистемой (3D audio)
- Попадание: звук при попадании по противнику
- Хэдшот: особый звук (dink)
- Смерть: звук при убийстве
- Перезарядка: звук перезарядки для каждого оружия
- UI: звуки нажатий кнопок, начала раунда, конца раунда

**Пространственный звук:**
- Шаги и выстрелы противников слышны с учётом расстояния и направления
- Громкость уменьшается с расстоянием
- Не требуется высокая точность (не как в CS2), но достаточная для ориентирования

**Технические требования:**
- Web Audio API для пространственного звука
- Форматы: OGG/MP3 (маленький размер файлов)
- Общая громкость + отдельные ползунки (эффекты, шаги)

**Acceptance Criteria:**
- Звуки выстрелов и шагов воспроизводятся с задержкой < 50 мс
- Пространственный звук корректно отражает позицию источника
- Можно отключить звук целиком или по категориям

## 3. Рекомендации по техническому стеку

### 3.1 Клиентская часть (Frontend)

| Компонент | Технология | Обоснование |
|---|---|---|
| 3D-движок | **Three.js** | Самая зрелая WebGL-библиотека, огромное комьюнити, отличная документация. Хорошо подходит для low-poly FPS |
| UI-фреймворк | **HTML/CSS + Vanilla JS** или **React** (для лобби) | Лобби и HUD можно сделать на обычном HTML поверх Canvas. React — опциональный вариант для лобби, но не обязателен |
| Сборщик | **Vite** | Быстрая сборка, HMR, отличная поддержка Three.js |
| Язык | **TypeScript** | Типизация поможет в большом проекте, особенно для сетевого кода |
| Хостинг клиента | **Netlify** | Как указано в требованиях. Подходит для статических файлов клиента |

### 3.2 Серверная часть (Backend) — ОБЯЗАТЕЛЬНА

> ⚠️ **Критически важно:** Для мультиплеерного шутера в реальном времени серверная часть необходима. Netlify может хостить только клиент. Для игрового сервера нужно отдельное решение.

| Компонент | Технология | Обоснование |
|---|---|---|
| Игровой сервер | **Colyseus** (Node.js) | Специализированный фреймворк для мультиплеерных игр. Встроенная система комнат, матчмейкинг, синхронизация состояния. Отличная документация и бесплатный open-source |
| Транспорт | **WebSocket** (через Colyseus) | Двунаправленная связь с низкой задержкой. Colyseus использует WebSocket под капотом |
| Хостинг сервера | **Colyseus Cloud** (бесплатный план) / **Railway** / **Render** / **Fly.io** | Бюджетные варианты хостинга Node.js сервера |

**Почему Colyseus, а не чистый Socket.IO:**
- Встроенная система комнат (rooms) — идеально для игровых лобби
- Авторитарная синхронизация состояния (state synchronization)
- Встроенная сериализация и дельта-обновления
- Tick-based game loop из коробки
- Активное комьюнити и документация

### 3.3 Архитектура сети

```
┌────────────────────┐         WebSocket          ┌─────────────────────┐
│   Client (Netlify) │ ◄──────────────────────── │  Game Server         │
│                    │                            │  (Colyseus/Node.js)  │
│  - Three.js        │  ────────────────────────► │                     │
│  - Input capture   │     Inputs (keystrokes,    │  - Game state        │
│  - Rendering       │     mouse movements)       │  - Physics           │
│  - Sound           │                            │  - Hit detection     │
│  - Prediction      │  ◄──────────────────────── │  - Round management  │
│                    │     State updates           │  - Room management   │
│                    │     (positions, HP, events) │                     │
└────────────────────┘                            └─────────────────────┘
      Netlify                                     Colyseus Cloud / VPS
```

**Сетевая модель:**
- **Authoritative Server:** Сервер — единственный источник правды. Все расчёты урона, хитдетекции и физики — на сервере
- **Client-side Prediction:** Клиент предсказывает результат своих действий (движение) для плавности
- **Server Reconciliation:** Сервер корректирует клиента, если предсказание расходится с реальностью
- **Entity Interpolation:** Позиции других игроков интерполируются между серверными обновлениями
- **Tick Rate:** 20–30 тиков/сек (баланс между точностью и нагрузкой)
- **Lag Compensation:** Сервер учитывает пинг игрока при хитдетекции (rewinding)

### 3.4 Оценка затрат

| Ресурс | Стоимость | Примечание |
|---|---|---|
| Netlify (клиент) | **Бесплатно** | Free tier: 100 GB bandwidth/мес |
| Colyseus Cloud (сервер) | **Бесплатно** (Free tier) или ~$10–20/мес | Free tier поддерживает ограниченное количество CCU |
| Альтернатива: Railway/Render | **Бесплатно** (с ограничениями) или ~$5–7/мес | Free tier с лимитом часов/месяц |
| Звуковые ассеты | **Бесплатно** | freesound.org, OpenGameArt (CC0 лицензия) |
| 3D-модели оружия | **Бесплатно** | Sketchfab (CC), OpenGameArt, или собственные low-poly модели в Blender |
| Домен (опционально) | ~$10–15/год | Если нужен свой домен вместо *.netlify.app |

**Итого MVP:** $0–20/мес (возможно полностью бесплатно на начальном этапе)

## 4. Концептуальная модель данных

### 4.1 Серверное состояние (Colyseus State Schema)

```
GameRoom
├── roomCode: string              // "A3F8K2"
├── status: enum                  // "lobby" | "weapon_select" | "playing" | "round_end" | "match_end"
├── settings
│   ├── mode: enum                // "1v1" | "2v2"
│   ├── mapId: string             // "warehouse" | "dust_alley" | "office" | "trainyard"
│   ├── roundsToWin: number       // 5, 7, 10, 13
│   └── roundTimeLimit: number    // секунды (например, 120)
├── adminId: string               // session ID создателя комнаты
├── scoreTeamA: number
├── scoreTeamB: number
├── currentRound: number
├── roundTimer: number
├── players: Map<string, Player>
│   └── Player
│       ├── sessionId: string
│       ├── nickname: string
│       ├── team: enum            // "A" | "B" | "unassigned"
│       ├── isAlive: boolean
│       ├── hp: number            // 0–100
│       ├── position: { x, y, z }
│       ├── rotation: { yaw, pitch }
│       ├── currentWeapon: enum   // "deagle" | "ssg08" | "mp9"
│       ├── ammo: number
│       ├── isReloading: boolean
│       ├── kills: number
│       └── deaths: number
└── killFeed: Array<KillEvent>
    └── KillEvent
        ├── killerNickname: string
        ├── victimNickname: string
        ├── weapon: string
        ├── isHeadshot: boolean
        └── timestamp: number
```

### 4.2 Клиент → Сервер (Input Messages)

| Сообщение | Поля | Описание |
|---|---|---|
| `move` | `{ keys: { w, a, s, d, space }, deltaTime }` | Нажатые клавиши движения |
| `look` | `{ yaw, pitch }` | Поворот камеры |
| `shoot` | `{ timestamp }` | Выстрел |
| `reload` | `{}` | Перезарядка |
| `selectWeapon` | `{ weapon: "deagle" \| "ssg08" \| "mp9" }` | Выбор оружия (в фазе выбора) |
| `joinTeam` | `{ team: "A" \| "B" }` | Выбор команды в лобби |
| `startGame` | `{}` | Админ начинает игру |

### 4.3 Сервер → Клиент (State Updates)

Colyseus автоматически синхронизирует изменения в State Schema через дельта-обновления. Дополнительные события:

| Событие | Данные | Описание |
|---|---|---|
| `hit` | `{ damage, isHeadshot, direction }` | Игрок получил урон |
| `kill` | `{ killer, victim, weapon, isHeadshot }` | Убийство произошло |
| `roundEnd` | `{ winnerTeam, scoreA, scoreB }` | Раунд завершён |
| `matchEnd` | `{ winnerTeam, finalScoreA, finalScoreB }` | Матч завершён |
| `playerDied` | `{ sessionId }` | Игрок погиб (для spectator mode) |
| `sound` | `{ type, position, sourceId }` | Звуковое событие для пространственного звука |

## 5. Принципы UI-дизайна

### 5.1 Общий стиль
- **Палитра:** Тёмные, приглушённые тона (тёмно-серый, чёрный) для UI, акценты — оранжевый/жёлтый (аналог CS2)
- **Шрифты:** Моноширинный или геометрический sans-serif (Rajdhani, Barlow Condensed)
- **Минимализм:** Минимум UI-элементов на экране во время игры
- **Адаптивность:** Основной таргет — десктопный браузер (не мобильные устройства)

### 5.2 Экраны

1. **Главный экран:**
   - Поле ввода никнейма
   - Две кнопки: "Создать комнату" / "Присоединиться"
   - Поле ввода кода комнаты (при присоединении)

2. **Лобби комнаты:**
   - Код комнаты (крупно, с кнопкой "Копировать")
   - Настройки матча (только для админа): карта, режим, раунды
   - Два столбца: Team A / Team B

## 6. Соображения по безопасности

### 6.1 Античит
- **Авторитарный сервер:** Все расчёты на сервере — клиент не может манипулировать HP, позицией или уроном
- **Валидация инпута:** Сервер проверяет, что игрок не двигается быстрее максимальной скорости
- **Rate limiting:** Ограничение частоты выстрелов на сервере (нельзя стрелять быстрее скорострельности оружия)
- **Server-side hit detection:** Попадания рассчитываются на сервере с учётом lag compensation

### 6.2 Сетевая безопасность
- WSS (WebSocket Secure) — шифрованное соединение
- Валидация всех входных данных на сервере
- Защита от спама сообщениями (throttling)
- Таймаут неактивных соединений

### 6.3 Защита от абьюза
- Никнеймы: фильтрация нецензурных слов (базовый фильтр)
- Комната автоматически уничтожается, если все покинули
- Код комнаты истекает через 10 минут без активности

## 8. Потенциальные проблемы и решения

### 8.1 Сетевая задержка (Latency)
**Проблема:** Высокий пинг делает игру неиграбельной для FPS.
**Решение:**
- Client-side prediction для мгновенного отклика на действия игрока
- Lag compensation на сервере (rewinding game state для проверки хитов)
- Entity interpolation для плавного отображения удалённых игроков
- Отображение пинга в UI, чтобы игрок понимал ситуацию

### 8.2 Производительность на слабых ПК
**Проблема:** WebGL может быть медленным на интегрированной графике.
**Решение:**
- Low-poly модели (< 500 полигонов на объект)
- Запечённое освещение (baked lightmaps) вместо реал-тайм теней
- Object pooling для пуль и эффектов
- Frustum culling (Three.js делает автоматически)
- Настройки качества: разрешение рендеринга, дальность прорисовки

### 8.3 Хитдетекция в сетевой игре
**Проблема:** Совместить авторитарный сервер с отзывчивостью стрельбы.
**Решение:**
- Клиент отправляет timestamp выстрела + направление
- Сервер отматывает состояние на timestamp с учётом пинга игрока
- Raycasting по позициям игроков на момент выстрела
- Визуальное подтверждение хита (hitmarker) приходит с сервера

### 8.4 Создание 3D-контента
**Проблема:** Нужны 3D-модели карт и оружия без опыта в моделировании.
**Решение:**
- Начать с prototype-карт из примитивов (кубы, цилиндры) в Three.js
- Использовать бесплатные low-poly ассеты: Kenney Assets (kenney.nl), OpenGameArt
- Blender (бесплатный) + обучающие YouTube-туториалы по low-poly моделированию
- GLTF как формат — отличная интеграция с Three.js

### 8.5 Масштабирование сервера
**Проблема:** Один сервер на Node.js ограничен по количеству комнат.
**Решение:**
- На старте: один сервер хватит для десятков одновременных комнат (2–4 игрока в каждой)
- При росте: Colyseus поддерживает горизонтальное масштабирование
- Colyseus Cloud автоматически масштабируется
